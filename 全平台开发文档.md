# Shelf 全平台开发文档

版本：v1.0（2026-07-08）
范围：Windows 单机版（v0.2.2）→ Windows / macOS / iPadOS / iOS 四端 + 账号云同步 + 各平台商店上架
关联文档：《技术设计文档.md》（单机版架构）、《PDF阅读器产品计划书_v1.0.md》（产品定位）

---

## 1. 目标与产品原则

### 1.1 目标

1. 同一本书在 iPad、iPhone、Mac、Windows 上随时接着读（进度、书架、元数据云同步）。
2. 软件可以分发给其他用户：注册账号即可多端同步。
3. 上架 App Store（iOS/iPadOS/macOS）与 Microsoft Store，官网提供直装包。

### 1.2 产品原则（架构决策的依据）

**原则一：本地优先，账号可选。**
不登录的 Shelf 等于今天的 Shelf——全功能、纯本地、零网络请求。登录后才解锁多端同步。
由此推导出的架构约束：同步是本地数据库之上的一层附加物，任何功能不得以"有网络/已登录"为前提。

**原则二：个人云书房，不做书籍分享。**
用户只能同步自己的文件，用户之间没有任何互传书籍的通道。
理由：一旦允许分享 PDF，产品性质从"工具"变为"内容平台"，需承担 UGC 审核义务，版权与商店审核风险陡增。

**原则三：一套代码，四个平台。**
继续使用 Tauri 2（已支持 Windows / macOS / iOS / Android），不做原生重写。
拒绝的备选方案：Swift 原生重写（Windows/Apple 双代码库永久双维护）、Flutter / React Native（前端与 PDF 渲染栈推倒重来）。

## 2. 现状盘点（v0.2.2）

| 项 | 现状 |
|---|---|
| 前端 | React 18.3 + TypeScript 5.6 + Vite 6，四个组件（Library / Reader / BookCover / WordPopup），约 1200 行 |
| 后端 | Rust（Tauri 2），单文件 `src-tauri/src/lib.rs` 约 550 行，8 个 Tauri command |
| PDF 渲染 | pdfjs-dist 4.10（WebView 内 Canvas 渲染） |
| 数据库 | rusqlite 0.32（bundled SQLite），书库元数据 + 阅读进度 |
| 词典 | 内置 ECDICT 离线词典（`resources/dict.db`，约 36 万词条） |
| 查重 | sha2 计算文件 SHA-256（**这是跨设备书籍身份的现成基础**） |
| 打包 | 仅 NSIS（Windows），`tauri.conf.json` targets = `["nsis"]` |

### 2.1 已知的跨平台障碍（阶段 0 必须清除）

1. **`lib.rs:515` 直接读 `APPDATA` 环境变量**，macOS/iOS 上不存在。改为 Tauri 的 `app.path().app_data_dir()`。
2. `tauri.conf.json` 中 `assetProtocol.scope` 为 `$DATA/Shelf/**`，需确认各平台展开路径正确。
3. 快捷键均为 Ctrl 系（Ctrl+滚轮缩放等），macOS 需映射 Cmd；移动端无键盘，交互需手势化。
4. 拖放添加 PDF 在移动端不存在，需接系统文件选择器与"分享到 Shelf"入口。
5. 打包目标只有 NSIS，需扩展 dmg / app / ipa / msix。

### 3.1 客户端（四端同一套）

| 层 | 选型 | 版本 | 说明 |

| PDF 渲染 | pdfjs-dist | 4.x | 沿用；移动端启用可见页渲染 + 降分辨率（见 §7.1） |
| 本地数据库 | rusqlite（bundled） | 0.32+ | 沿用；iOS 上 SQLite 为系统组件，bundled 亦可用 |
| 哈希 | sha2 | 0.10 | 沿用，SHA-256 为全局书籍 ID |
| 文件选择 | tauri-plugin-dialog | 2.x | 已引入，桌面与移动端均支持 |

| 密钥存储 | tauri-plugin-stronghold 或 keyring | — | 新增，存 refresh token（iOS Keychain / Windows 凭据管理器 / macOS Keychain） |

### 3.2 云端

| 层 | 选型 | 说明 |
|---|---|---|
| 账号认证 | Supabase Auth | 邮箱 + Sign in with Apple + Google；iOS 提供第三方登录时 Apple 登录为强制项 |
| 元数据库 | Supabase Postgres | 用户、书籍元数据、阅读进度、配额；行级权限（RLS）保证用户只能读写自己的行 |
| 文件存储 | Cloudflare R2 | PDF 与封面缩略图；出口流量免费是选它而非 S3 的决定性理由（见 §9.2） |
| 上传/下载 | R2 预签名 URL | 由一个轻量 Worker（或 Supabase Edge Function）签发，文件流量不经过应用服务器 |
| 同步接口 | PostgREST（Supabase 自带） | 元数据/进度直接走 Supabase REST + RLS，起步阶段不写自定义 API 服务 |

| 项 | 选型 |
|---|---|
| CI/CD | GitHub Actions（matrix：windows-latest / macos-latest；iOS 构建跑在 macos runner） |
| 桌面分发 | 官网 DMG（公证）+ NSIS；Microsoft Store（msix）；Mac App Store |
| 移动分发 | TestFlight → App Store |

## 4. 系统架构

```
┌────────────────────────────────────────────────────┐
│  客户端（Windows / macOS / iPadOS / iOS，同一套代码） │
│                                                    │
│  React UI ── invoke ──▶ Rust 核心                   │
│                         ├─ SQLite（唯一事实来源）     │
│                         ├─ 书库文件目录（app_data）   │
│                         └─ SyncEngine（后台任务）     │
└──────────────────────────────┬─────────────────────┘
                               │ 登录后启用，HTTPS
              ┌────────────────┴──────────────────┐
              │                                   │
      Supabase（Auth + Postgres + RLS）      Cloudflare R2
      账号 · 元数据 · 进度 · 配额              PDF 文件 · 封面
              └── Edge Function 签发预签名 URL ──┘
```

要点：

1. **SQLite 是唯一事实来源**。UI 永远读写本地库；SyncEngine 在后台把本地变更推到云端、把云端变更合并回本地。断网时一切照常，联网后追平。
2. **SyncEngine 在 Rust 侧实现**，做成 trait（`SyncBackend`），官方云为默认实现；WebDAV 可作为高级用户自托管后端的第二实现（非 MVP）。
3. **文件流量不经过服务器**：客户端向 Edge Function 请求预签名 URL，直接与 R2 收发文件。

## 5. 数据模型

### 5.1 本地 SQLite（在现有表上增列）

```sql
-- 现有 books 表增加同步字段
ALTER TABLE books ADD COLUMN updated_at INTEGER NOT NULL DEFAULT 0; -- Unix 毫秒
ALTER TABLE books ADD COLUMN deleted    INTEGER NOT NULL DEFAULT 0; -- 墓碑
ALTER TABLE books ADD COLUMN synced_at  INTEGER NOT NULL DEFAULT 0; -- 上次成功同步时间
ALTER TABLE books ADD COLUMN cloud_state TEXT NOT NULL DEFAULT 'local';
-- cloud_state: local（仅本机）| uploading | synced | remote（云端有、本机未下载）

-- 进度表同理增加 updated_at；device_id 存全局设置表
CREATE TABLE IF NOT EXISTS sync_meta (
  key TEXT PRIMARY KEY,   -- device_id / last_pull_cursor / user_id ...
  value TEXT NOT NULL
);
```

### 5.2 云端 Postgres（Supabase，全部启用 RLS：`user_id = auth.uid()`）

```sql
create table books (
  user_id     uuid    not null references auth.users(id) on delete cascade,
  sha256      text    not null,          -- 全局书籍 ID
  title       text    not null,
  author      text,
  page_count  int,
  file_size   bigint  not null,
  cover_key   text,                      -- R2 对象键，可空
  file_key    text,                      -- R2 对象键，可空（未上传正文时为 null）
  updated_at  timestamptz not null,
  deleted     boolean not null default false,
  primary key (user_id, sha256)
);

create table reading_progress (
  user_id     uuid not null references auth.users(id) on delete cascade,
  sha256      text not null,
  page        int  not null,
  zoom_mode   text,
  view_mode   text,                      -- single / double
  updated_at  timestamptz not null,
  device_name text,                      -- 冲突提示用："上次读到 P123（来自 iPad）"
  primary key (user_id, sha256)
);

create table user_quota (
  user_id     uuid primary key references auth.users(id) on delete cascade,
  bytes_used  bigint not null default 0,
  bytes_limit bigint not null default 536870912,  -- 免费档 512 MiB
  plan        text   not null default 'free'
);
```

### 5.3 R2 对象布局

```
{user_id}/books/{sha256}.pdf      -- 原始文件，内容寻址，天然免重复上传
{user_id}/covers/{sha256}.jpg     -- 封面缩略图，宽 480px，JPEG q80，约 40–80 KB
```

对象键包含 user_id 前缀，预签名 URL 签发时校验请求者身份与前缀一致，实现存储层隔离。

## 6. 同步协议

### 6.1 身份与合并策略

- **书籍身份 = 文件 SHA-256**（复用现有查重逻辑）。同一本书无论在哪台设备导入，哈希一致即同一本书，进度自动对齐。
- **合并策略 = LWW（最新时间戳获胜）**，粒度为"行"。个人多设备场景冲突极少且损失可容忍（最多回退几页进度），不引入 CRDT。
- **删除用墓碑**（`deleted = true`）而非物理删行，否则删除无法传播。墓碑保留 90 天后由云端定期清理。
- **时钟**：LWW 依赖设备时钟。进度合并额外规则——若两条记录时间戳差 < 2 分钟，取 `page` 较大者（防止设备时钟略偏导致进度回跳）。

### 6.2 同步流程（SyncEngine，Rust 后台任务）

```
触发时机：App 启动、翻页防抖 30s、书架变更、每 5 分钟心跳、网络恢复

push:
  取本地 updated_at > synced_at 的行
  → upsert 到 Supabase（携带 updated_at）
  → 云端 RLS 校验归属；成功后写回 synced_at

pull:
  按 last_pull_cursor（云端 updated_at 游标）增量拉取
  → 逐行与本地比较 updated_at，云端新则覆盖本地，本地新则留待下轮 push
  → 更新 last_pull_cursor

文件上传（书籍入库后异步）:
  配额检查 → Edge Function 签发预签名 PUT → 直传 R2
  → 成功后更新 books.file_key、user_quota.bytes_used、cloud_state='synced'
  大文件（>100 MB）走 R2 分片上传，断点续传

文件下载（打开 remote 书籍时）:
  签发预签名 GET → 下载到书库目录 → 校验 SHA-256 → cloud_state='synced'
```

### 6.3 进度上报的流量控制

翻页事件防抖 30 秒合并为一次上报（一条 ~1 KB 的 upsert）。重度用户每天约 100 次上报 ≈ 100 KB/天，流量成本可忽略（见 §9.3）。

## 7. 平台适配要点

### 7.1 iOS / iPadOS（工作量最大的一端）

- **内存预算（这是移动端最硬的约束）**：WKWebView 进程超限会被系统直接杀掉（iPhone 上限约 1–1.5 GB）。渲染位图内存 = 宽 × 高 × 4 字节：
  - A4 页按 300 DPI 全质量渲染：2480 × 3508 × 4 ≈ **35 MB/页**，同屏 3 页即超 100 MB，加上 PDF.js 解析结构极易触顶——**禁止**。
  - 按设备物理分辨率渲染（iPhone 15 Pro 宽 1179px）：1179 × 1668 × 4 ≈ **7.9 MB/页**，可见页 ±1 共 3 页 ≈ 24 MB——**目标方案**。
  - 实施：页面虚拟化（只保留可见页 ±1 的 Canvas，其余销毁）、渲染尺寸 = 视口尺寸 × devicePixelRatio（封顶 2）、缩放结束再重渲染（缩放过程中用 CSS transform 拉伸旧位图）。
  - 兜底：若扫描版巨型 PDF 仍崩，在 Rust 侧用 pdfium-render 生成页面位图流给前端（v2 预案，不在 MVP）。
- 触屏交互：左右滑动/点击两侧翻页、捏合缩放、长按取词（替代双击）、无 hover 依赖、点击目标 ≥ 44pt、适配安全区（刘海/圆角）。
- 文件导入：`tauri-plugin-dialog` 文件选择器 + 注册 PDF 文档类型（Info.plist `CFBundleDocumentTypes` + `UTImportedTypeDeclarations`），支持从"文件"App 和其他 App"分享到 Shelf"。
- 生命周期：进入后台立即持久化进度并触发一次 push（iOS 随时可能杀后台进程）。
- 词典 dict.db（几十 MB）作为 bundle 资源随包分发，首启动拷贝到 app_data。

### 7.2 macOS

- Cmd 系快捷键映射；原生菜单栏（About / 偏好设置 / 退出）；红绿灯窗口按钮不遮挡 UI。
- 打包：官网渠道 DMG + Apple 公证（notarization）；商店渠道 Mac App Store（需 App Sandbox，文件访问经用户选择自动获得例外，Shelf 模式天然兼容）。

### 7.3 Windows（存量）

- 行为不变；新增账号/同步 UI 与其他端一致。
- 官网安装包接入 Azure Trusted Signing 消除 SmartScreen 拦截；Microsoft Store 打 msix。

## 8. 开发流程

### 8.1 分支与发布

- 主干开发（trunk-based）：`main` 始终可构建；功能走短生命周期分支 + PR。
- 版本号统一由 `tauri.conf.json` 驱动，`git tag v*` 触发发布流水线。
- 发布节奏：桌面端随时发（自动更新）；iOS 攒批走 TestFlight → App Store 审核。

### 8.2 CI/CD（GitHub Actions）

| 工作流 | 触发 | 内容 |
|---|---|---|
| ci.yml | 每次 PR | `tsc --noEmit`、`cargo clippy`、`cargo test`（重点覆盖同步合并逻辑）、双平台 debug 构建 |
| release-desktop.yml | tag | Windows NSIS + 签名；macOS universal DMG + 公证；上传 GitHub Release + 更新源 |
| release-ios.yml | tag | `tauri ios build` → ipa → fastlane 上传 TestFlight |

### 8.3 测试策略

- **单元测试（必须）**：同步合并逻辑（LWW、墓碑、时钟偏移规则）是全项目最不能出错的代码，用 Rust 单测覆盖全部分支；SHA-256 入库/查重回归。
- **集成测试**：对接 Supabase 本地栈（`supabase start`）跑真实 push/pull 循环；双客户端模拟"设备 A 翻页 → 设备 B 拉取"。
- **手动清单**：每次发布前过一遍四端核心路径（导入 → 阅读 → 翻页 → 换设备接着读 → 删除传播）。
- **内存验证**：iOS 端用 Instruments 压测 300 MB+ 扫描版 PDF。

### 8.4 环境

- Supabase 两个项目：`shelf-dev` / `shelf-prod`；R2 两个 bucket 同理。客户端按构建配置切换。
- 密钥（签名证书、Supabase service key、R2 token）全部进 GitHub Actions Secrets，仓库零明文。

## 9. 成本与容量测算

### 9.1 基础假设

| 参数 | 取值 | 依据 |
|---|---|---|
| 单本 PDF 平均体积 | 40 MB | 文字版 5–20 MB，扫描版 50–300 MB，按混合库估 |
| 人均云端藏书 | 25 本 | 重读型个人书库 |
| **人均存储** | **≈ 1 GB** | 25 × 40 MB |
| 封面缩略图 | 60 KB/本 | 480px JPEG q80，25 本 ≈ 1.5 MB，可忽略 |
| 进度上报 | 1 KB × 100 次/天（重度） | 30s 防抖后 |

### 9.2 变动成本（随用户数线性）

R2 定价：存储 $0.015/GB·月；写（Class A）$4.50/百万次；读（Class B）$0.36/百万次；**出口流量 $0**。

| 用户规模 | 存储 | 操作费 | R2 月成本 |
|---|---|---|---|
| 100 人 | 100 GB × $0.015 = $1.5 | 忽略不计* | ≈ $1.5 |
| 1 000 人 | 1 TB → $15 | < $0.1 | ≈ $15 |
| 10 000 人 | 10 TB → $150 | < $1 | ≈ $150 |

\* 操作费估算：每用户每月上传 5 本（5 次 Class A）+ 打开/下载 30 次（Class B）→ 1000 用户 = 5 000 写 + 30 000 读 ≈ $0.02 + $0.01。

**每免费用户边际成本 ≈ 1 GB × $0.015 = $0.015/月（约 1 毛钱人民币）**——这是配额与商业模式设计的锚点。

对比：若用 S3（出口 $0.09/GB），1000 用户每月哪怕只重下 10% 的库，流量费就是 100 GB × $0.09 = $9，且随换机、重装线性放大。**出口免费是选 R2 的决定性理由。**

元数据侧：books 行约 0.5 KB、进度行约 0.2 KB → 1000 用户 × 25 本 ≈ 17 MB，Supabase Free 档（500 MB 库、5 万 MAU）足够支撑到几千用户；上 Pro（$25/月）主要为了每日备份 + 项目不休眠，建议公测前升级。

### 9.3 固定成本

| 项 | 费用 | 折合每月 |
|---|---|---|
| Apple Developer Program | $99/年 | $8.3 |
| Microsoft 个人开发者 | $19 一次性 | ≈ 0 |
| Azure Trusted Signing | $9.99/月 | $10 |
| Supabase Pro | $25/月 | $25 |
| 域名 + 静态官网 | ~$12/年 | $1 |
| **合计** | | **≈ $44/月** |

### 9.4 配额与订阅定价推算

| 档位 | 配额 | 存储成本 | 定价建议 | 毛利 |
|---|---|---|---|---|
| Free | 512 MB（约 12 本文字版） | $0.008/月 | ¥0 | — |
| Pro | 20 GB | $0.30/月 ≈ ¥2.2 | ¥15/月 或 ¥128/年 | iOS 内购扣 15% 后到手 ¥12.75，毛利 > 80% |

**盈亏平衡**：固定成本 $44/月 ÷ 每 Pro 订户净收入 ≈ $1.8/月 → **约 25 个付费订户覆盖全部固定成本**。此后每新增 1 个免费用户仅增加 $0.015/月，一个 Pro 订户可养活约 120 个免费用户。

结论：即使完全不追求盈利，配额机制也必须从第一天设计进去——它是存储成本的闸门；定价上 ¥15/月有充足安全边际。

### 9.5 iOS 渲染内存预算（复述硬约束）

`页面位图内存 = 宽px × 高px × 4B`。目标：**渲染分辨率 = 视口 × min(devicePixelRatio, 2)，常驻 Canvas ≤ 3 页，总渲染内存 < 30 MB**（iPhone 15 Pro：1179 × 1668 × 4 × 3 ≈ 24 MB）。任何改动 Reader 渲染逻辑的 PR 都不得突破此预算。

## 12. 风险登记

| # | 风险 | 等级 | 对策 |
|---|---|---|---|
| 1 | iPhone 大 PDF 内存超限被杀 | 高 | §7.1 渲染预算硬约束；pdfium 位图方案为兜底预案 |

| 5 | LWW 极端场景进度回跳 | 低 | 时钟偏移保护（§6.1）；进度冲突时 UI 提示"iPad 读到 P123，跳转？" |
| 6 | Supabase 供应商锁定 | 低 | 全部是标准 Postgres + GoTrue，可整体迁自托管；SyncBackend trait 保留切换口 |
| 7 | App Store 审核被拒 | 低 | 工具类 + 无 UGC 分享 + 审核员免登录可体验核心功能 |
